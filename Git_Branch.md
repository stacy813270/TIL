## Git Branch

### Branch Command
- master 외에도 Branch 있음
- master(원본)에서 개발 진행하지 않음
- 예를 들어 로그인 기능 브렌치, 게시판 기능 브렌치 등 따로 만듦
- 명령어 한 줄로 브렌치 생성, 이동 간편
- 사용 이유: 상용 중인 서비스에 오류가 났을 때 그 서비스를 복제한 후 거기에 테스트하고 다시 원본에 머지해야 함. master Branch에는 확정된 안정적인 코드만 포함. 실무에서도 팀장급이 포함 결정
- 브랜치 조회, 생성, 삭제 등 브랜치와 관련된 git 명령어
  - git branch
  - 병합된 브랜치는 역할을 다했기 때문에 삭제됨
  - git switch
  - head: 현재 브랜치나 commit을 가리키는 포인터. 현재 내가 보고있는 위치
  - git switch 주의사항
    - 만든 순서와 의존 화살표 순서가 다름
    - 피처에서 스위치 시에 커밋을 안 찍고 마스터로 이동하면 피처에서 만들고 있는 파일이 마스터에도 존재하게 됨. 그러면 개발이 꼬이는 것.
    - 스위치 하기 전에 git status로 확인 하고 add가 안된 파일이 있는지 확인해야 함
  - git merge: 두 브랜치를 하나로 병합(결합)
    - 주의사항 - 수신 브랜치 확인, 최신 머지 상태 확인
    - Fast-Forward Merge: 마스터에서 분할된 브랜치에서 수정한 후에 기존 마스터에 변화가 없다면 분할된 브랜치가 가장 최신이다. 이때 힘든 병합 과정을 하지 않고 마스터를 분할된 브랜치(최신 상태)로 땡겨오는 것
    - 3-Way Merge - 머지 커밋이 생김
    - 'vim' 에디터들의 조상. 마우스가 없었던 시절. 가장 가벼운 에디터. vim 배열 키보드(해피해킹. 흑무각 간지남)
      - vim 모드 2가지
        - 입력 모드(텍스트 입력(i))
        - 명령 모드(: + 저장(w), 종료(q)...)
    - Merge Conflict: '동일한 파일의 동일한 부분'. 동일한 파일에서 다른 부분을 수정했으면 충돌 안 남
